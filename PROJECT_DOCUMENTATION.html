1. Project Overview & Motivation

Every SaaS company, e-commerce platform, and service provider needs a system to track customer issues — from initial report to resolution. I built this Customer Support Ticket System to solve that exact problem: a platform where customers submit support requests and agents manage, respond to, and resolve them.

What Problem Does This Solve?

  • Customers can create tickets, track their status, and communicate with support agents — all in one place
  • Support Agents get a centralized dashboard to manage all tickets, assign workload, and use AI-assisted replies
  • Organizations get visibility into support metrics — open/in-progress/resolved counts, response times

Key Features I Implemented

  • Role-based access control (Customer vs Agent) with distinct UI and permissions
  • Complete ticket lifecycle: Open → In Progress → Resolved with transition validation
  • JWT-based authentication with bcrypt password hashing
  • 16+ RESTful API endpoints with auto-generated Swagger documentation
  • RAG-powered AI reply suggestions — Groq (Llama 3.3 70B) grounded in past resolved tickets
  • Qdrant vector DB — semantic similar ticket search (meaning, not keywords)
  • DistilBERT priority classifier — 94% accuracy, returns softmax confidence score
  • LangChain LCEL — ChatPromptTemplate | ChatGroq | StrOutputParser chain
  • Category system, priority levels, search & filtering
  • Responsive SPA frontend with modern UI/UX
Why This Project? I chose to build a ticketing system because it mirrors real-world SaaS tools (Zendesk, Freshdesk, Jira Service Desk) and demonstrates full-stack competency: authentication, database design, REST API architecture, state management, and UI development — all in one project.

4. Database Schema Design

I designed a normalized relational schema with four tables and indexed foreign-key relationships. The schema follows 3NF (Third Normal Form) to eliminate data redundancy.

┌──────────────────┐ ┌──────────────────┐ │ users │ │ categories │ ├──────────────────┤ ├──────────────────┤ │ id (PK) │ │ id (PK) │ │ name │ │ name (UNIQUE) │ │ email (UNIQUE) │ ┌───→│ description │ │ password_hash │ │ └──────────────────┘ │ role │ │ │ is_active │ │ │ created_at │ │ └────┬─────┬───────┘ │ │ │ │ │ │ ┌────────┴─────────────┐ │ │ │ tickets │ │ │ ├──────────────────────┤ │ ├───→│ id (PK) │ │ │ │ title │ │ │ │ description │ │ │ │ status (indexed) │ │ │ │ priority (indexed) │ │ └───→│ customer_id (FK) │ │ │ agent_id (FK) │ │ │ category_id (FK) ────┘ │ │ created_at │ │ │ updated_at │ │ └────────┬─────────────┘ │ │ │ ┌────────┴─────────────┐ │ │ comments │ │ ├──────────────────────┤ │ │ id (PK) │ └─────────→│ user_id (FK) │ │ ticket_id (FK, CASCADE)│ │ content │ │ is_ai_generated │ │ created_at │ └──────────────────────┘

Design Decisions

  • Two FK on tickets → users: customer_id (who created it) and agent_id (who's handling it). This models the real-world assignment workflow.
  • Cascade delete on comments: When a ticket is deleted, its comments are automatically removed.
  • Indexed columns: status, priority, customer_id, and agent_id are indexed for fast filtering on the dashboard.
  • is_ai_generated flag: Comments created via AI suggestion are marked, providing transparency.
  • Soft delete pattern: is_active on users allows deactivation without data loss.

FastAPI Application Structure

The backend follows a modular router pattern. Each domain (auth, tickets, comments, etc.) has its own router file with dedicated endpoints. This keeps the codebase organized and each file under 200 lines.

Request Validation with Pydantic

Every endpoint uses Pydantic schemas for input validation. FastAPI automatically returns 422 errors for invalid data:

    description: str = Field(..., min_length=10)

Status Transition Validation

I enforce valid ticket status transitions to prevent inconsistent states:

valid_transitions = {
    "open": ["in_progress", "resolved"],
    "in_progress": ["open", "resolved"],
    "resolved": ["open"],  # Can reopen resolved tickets
}

if new_status not in valid_transitions[current_status]:
    raise HTTPException(400, "Invalid status transition")

Role-Based Access Control

# Dependency factory for role checking
def require_role(allowed_roles: list):
    def role_checker(current_user = Depends(get_current_user)):
        if current_user.role not in allowed_roles:
            raise HTTPException(403, "Forbidden")
        return current_user
    return role_checker

# Usage: Only agents can assign tickets
@router.patch("/{ticket_id}/assign")

    ...

Security Measures

  • bcrypt hashing: Adaptive work factor protects against brute-force attacks
  • CORS configuration: Only configured origins can access the API
  • Input validation: Pydantic schemas reject malformed input before it reaches business logic
  • Row-level security: Customers can only see their own tickets (enforced in query filters)
  • No sensitive data in JWT: Token only contains user_id and role — not email or password

8. AI Intelligence Stack (v2.0)

The AI layer was completely rebuilt in v2.0. It now implements a full RAG (Retrieval-Augmented Generation) pipeline backed by a fine-tuned vector search engine and a DistilBERT priority classifier.

Layer 1 — Qdrant Semantic Search

When a ticket arrives, its text is embedded using a fine-tuned MiniLM model and stored in Qdrant. The similar-ticket endpoint queries Qdrant using HNSW cosine search — finding tickets by meaning, not keywords.

# similarity_engine.py
results = client.query_points(
    collection_name="tickets",
    query=embedding_vector,
    query_filter=Filter(must=[FieldCondition(key="status", match=MatchValue(value="resolved"))]),
    limit=3,
)
# Returns ScoredPoint objects with payload + similarity score

Layer 2 — RAG Reply Generation (LangChain LCEL)

The top resolved similar tickets are used as context. A LangChain LCEL chain passes this context to Groq (Llama 3.3 70B) to generate a contextual, grounded reply.

# routers/ai_suggest.py
chain = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful support agent. Use the resolved ticket context below."),
    ("human", "Context:\n{context}\n\nCurrent ticket: {ticket}\n\nWrite a reply:"),
]) | ChatGroq(model="llama-3.3-70b-versatile") | StrOutputParser()

reply = chain.invoke({"context": rag_context, "ticket": ticket_text})

Layer 3 — DistilBERT Priority Classifier

A fine-tuned distilbert-base-uncased model classifies each ticket as low, medium, or urgent. Trained on 1,071 tickets using the HuggingFace Trainer API with LLM-assisted auto-labelling via Groq.

# Inference in similarity_engine.py
logits = model(**inputs).logits
probs = torch.softmax(logits, dim=-1)
confidence = float(probs[0].max())      # e.g. 0.97
label = ID2LABEL[int(probs[0].argmax())]  # e.g. "urgent"

Fallback Chain

Always returns a response The system degrades gracefully: RAG+Groq → LLM only (no context) → Static templates. The method field in the response tells you which path fired.

11. Challenges Faced & Solutions

ChallengeSolution
N+1 query problem — Loading tickets with customer/agent names caused hundreds of SQL queries Used SQLAlchemy's joinedload to eagerly fetch related data in a single query. Dashboard load time dropped from ~800ms to ~50ms.
JWT token management — Keeping auth state synced between API client and React context Axios interceptors handle token attachment and 401 cleanup. AuthContext initializes by calling /api/auth/me on app load to validate stored tokens.
Status transition validation — Preventing invalid state changes (e.g., resolved → in_progress directly) Defined a transition map and validated before each update. Returns clear error messages for invalid transitions.
CORS configuration — React dev server on port 5173, FastAPI on 8000 Configured CORS middleware with the frontend URL. Used Vite's proxy in development for seamless API calls.
Role-based UI — Same app, different capabilities for customers vs agents AuthContext exposes user.role. Components conditionally render features (e.g., "Assign to me" button only for agents). Backend enforces access at the API level too.
Database portability — Need SQLite for easy setup, PostgreSQL for production SQLAlchemy ORM abstracts the database. Only the connection string changes between SQLite and PostgreSQL.

15. Technical Architecture FAQ

Architecture Notes This section compiles system design decisions, architecture details, and answers to common technical queries regarding the implementation of the ticketing system.

📌 Project Overview & Motivation

Q: Tell me about this project in 2 minutes.
A: I built a full-stack AI-powered Customer Support Ticket System — think a production-grade Zendesk clone with a full ML intelligence layer. Customers log in, create support tickets, and track status. Agents get a centralized dashboard with AI-assisted replies powered by RAG.

The v2.0 ML stack has 5 phases: (1) Qdrant vector database replaces NumPy brute-force search — semantic similarity, not keyword matching. (2) RAG pipeline — when an agent clicks "Suggest Reply", the system finds similar resolved tickets in Qdrant, pulls their resolution comments, and injects that as context into (3) a LangChain LCEL chain calling Groq's Llama 3.3 70B — completely free. (4) Fine-tuned MiniLM for domain-specific embeddings. (5) DistilBERT classifier trained on 1,071 tickets (121 real + 950 LLM-generated synthetic data) — 94% accuracy on priority prediction with softmax confidence scores.

Tech: React + Vite frontend, FastAPI backend, SQLAlchemy + SQLite, HuggingFace Transformers, LangChain, Qdrant. Deployed on Vercel (frontend) + Render (backend).
Q: Why did you build this specific project?
A: I wanted one project that demonstrates everything a full-stack role requires: database design, REST API architecture, authentication, role-based authorization, state management in React, and cloud deployment. A ticketing system is perfect because it has a real-world business context — companies like Zendesk and Freshdesk make millions solving exactly this problem. It is a complete, deployable demonstration of clean REST APIs and database patterns.
Q: Have you used this in production? Is it live?
A: Yes — the frontend is live on Vercel at customer-support-ticket-system-lime.vercel.app and the backend API is deployed on Render. The backend uses SQLite in the free tier deployment. If this were a real production system, I'd use Render's managed PostgreSQL or a hosted DB like Supabase, and add proper DB backups and monitoring.

🏗️ Architecture & Design

Q: Why did you separate the frontend and backend? Why not a monolith?
A: Decoupled architecture gives three main advantages: independent deployment — I can push a frontend fix without restarting the backend; scalability — if the API gets heavy traffic, I can scale just the backend; and API reusability — if we later build a mobile app, it consumes the same FastAPI backend without any changes. The trade-off is more complexity in CORS configuration and separate deployment pipelines, but for any real-world SaaS product, this is the standard approach.
Q: How did you design the database schema?
A: I followed 3NF (Third Normal Form) normalization. There are four tables: users, categories, tickets, and comments. The key design decision was putting two foreign keys on the tickets table referencing users — one for customer_id (who created it) and one for agent_id (who's handling it). This models the real-world assignment workflow cleanly. I also added database indexes on status, priority, customer_id, and agent_id because those are the most-filtered columns on the dashboard query.
Q: What is the ticket lifecycle? How do status transitions work?
A: Tickets follow Open → In Progress → Resolved, with the option to reopen a resolved ticket back to Open. I enforce this in the backend with a transition map: 
valid_transitions = {
    "open":        ["in_progress", "resolved"],
    "in_progress": ["open", "resolved"],
    "resolved":    ["open"],  # can reopen
}
If the new status isn't in the allowed list, the API returns HTTP 400 with a clear error message. This prevents data inconsistencies — for example, you can't jump from open directly to resolved without the agent picking it up (though I do allow it for flexibility). The status change also auto-promotes a ticket to in_progress when an agent assigns it.

⚙️ Tech Stack Decisions

Q: Why FastAPI and not Flask or Django?
A: Three reasons. First, FastAPI gives you automatic Swagger UI at /docs — zero extra work for API documentation. Second, it uses Pydantic for request validation, so I define the schema once and get type checking, serialization, and auto-generated docs all from the same class. Third, FastAPI is async-capable (ASGI), so it's more performant than Flask (WSGI) for I/O-bound workloads like database queries. Django would have been overkill — it comes with an ORM, admin, templating engine, and auth system I'd be replacing anyway.
Q: Why SQLAlchemy? Why not raw SQL or another ORM?
A: SQLAlchemy is Python's most mature ORM and abstracts the database engine, so my code runs on both SQLite (development) and PostgreSQL (production) with only a connection-string change. The most important feature I used was joinedload to solve the N+1 query problem — without it, loading 20 tickets with customer names and agent names would fire 41 separate SQL queries. With joinedload, it's a single JOIN query.
Q: Why React? Why not plain HTML/JS or Vue?
A: The dashboard has dynamic state — filters, live ticket updates, role-based conditional rendering, and form state management. React's component model and virtual DOM make managing this complexity clean and predictable. Context API handles global auth state without needing Redux for an app of this size. React also has the largest ecosystem and is the most-asked-for skill in frontend job listings, so it was the logical choice for a portfolio project.
Q: Why Vite instead of Create React App?
A: Vite's dev server starts in under a second and uses native ES modules, giving instant Hot Module Replacement. Create React App uses Webpack, which has 30–60 second cold starts on larger projects. For development productivity, Vite is simply better. CRA is also no longer actively maintained by the React team.

🔐 Authentication & Security

Q: How does JWT authentication work in your system?
A: When a user logs in, the backend verifies their credentials, then issues a signed JWT token containing the user's ID, role, and expiry time. The frontend stores this token in localStorage and attaches it to every API request via an Axios interceptor in the Authorization: Bearer <token> header. On each protected endpoint, FastAPI's get_current_user dependency decodes and verifies the token. If it's expired or invalid, the API returns 401, and the Axios interceptor clears the stored token and redirects the user to the login page.
Q: Why JWT instead of session-based auth?
A: JWT is stateless — the server doesn't store session data in memory or a database. This is important for scalability: if I had multiple backend instances behind a load balancer, session-based auth would require a shared session store (like Redis). With JWT, any instance can verify the token by checking the cryptographic signature. The trade-off is that JWTs can't be invalidated before expiry — but I mitigated this by setting a short 60-minute expiry.
Q: Is storing JWT in localStorage safe?
A: This is a legitimate security question. localStorage is vulnerable to XSS attacks — if an attacker injects malicious JavaScript, they can read the token. The more secure alternative is storing the JWT in an HttpOnly cookie, which JavaScript can't access. For this project, localStorage was acceptable because: (1) it's a demo/portfolio project without real sensitive data, (2) the API validates tokens on every request, and (3) the CORS policy restricts which origins can call the API. In a real production app, I'd use HttpOnly cookies.
Q: How does role-based access work?
A: There are two roles: customer and agent. Enforcement happens at two levels. At the API level, I use a require_role() dependency factory — endpoints like "assign ticket" or "AI suggest" only accept requests from users with the agent role; otherwise they return HTTP 403. At the data level, customers can only query their own tickets — the list_tickets endpoint filters by customer_id == current_user.id for non-agents. Frontend-side, the user.role from AuthContext conditionally shows/hides UI elements, but I never rely solely on the frontend for security.
Q: How are passwords stored?
A: Passwords are hashed using bcrypt before being stored. bcrypt is an adaptive hashing algorithm — it has a configurable "work factor" that makes it intentionally slow to compute, which protects against brute-force attacks. Even if the database is compromised, the attacker can't reverse the hash. Plaintext passwords are never logged or stored anywhere in the system.

⚡ Performance & Scalability

Q: What is the N+1 query problem and how did you solve it?
A: The N+1 problem occurs when you run 1 query to fetch a list, then N additional queries to fetch related data for each item. For example, loading 20 tickets and then querying the customer name for each ticket = 21 queries. I solved this using SQLAlchemy's joinedload(): 
db.query(Ticket).options(
    joinedload(Ticket.customer),
    joinedload(Ticket.agent),
    joinedload(Ticket.category),
)
This generates a single SQL query with JOINs. The dashboard load time went from ~800ms to ~50ms as a result.
Q: How does pagination work?
A: The GET /api/tickets/ endpoint accepts page and limit query parameters. Pagination is implemented with SQL OFFSET and LIMIT: 
query.offset((page - 1) * limit).limit(limit).all()
The default is 20 tickets per page with a maximum of 100. This prevents loading thousands of rows into memory on a large dataset. The frontend sends ?page=1&limit=20 and can request the next page by incrementing the page number.
Q: What database indexes did you add and why?
A: I indexed status, priority, customer_id, and agent_id on the tickets table. These are the columns used in the most common WHERE clauses: "show all open tickets", "show my assigned tickets", "filter by priority". Without indexes, each of these would require a full table scan. With indexes, the database can look up matching rows in O(log n) time using a B-tree structure. Email on users is also indexed for fast login lookups.

🤖 AI Feature

Q: Tell me about the AI reply suggestion feature.
A: When an agent is viewing a ticket, they can click "Suggest Reply" which calls POST /api/ai/suggest-reply. The system works in two modes. By default it uses a template-based system — it looks at the ticket's category (billing, technical, account, general) and returns a contextually appropriate pre-written response, randomized from a pool to avoid repetition. If an OPENAI_API_KEY is configured as an environment variable, it switches to GPT-3.5-turbo — it constructs a prompt with the ticket title, description, category, priority, and recent comments, then calls the OpenAI API. If the OpenAI call fails for any reason (rate limit, network error), it silently falls back to templates, so the feature always works.
Q: Why template-based and not always OpenAI?
A: Cost and reliability. OpenAI API calls cost money and have rate limits. For a demo or small deployment, templates provide instant, free suggestions that work offline. It also makes the project deployable without needing anyone to have or pay for an API key. The system is designed so upgrading to real AI is just adding an environment variable — no code changes needed.

🚀 Deployment

Q: How did you deploy this project?
A: The frontend is deployed on Vercel — I connected my GitHub repo, set the root directory to frontend/, added the VITE_API_URL environment variable pointing to my Render backend, and Vercel handles build, CDN distribution, and HTTPS automatically on every git push. The backend is deployed on Render — I configured a Web Service with root directory backend/, build command pip install -r requirements.txt, and start command uvicorn main:app --host 0.0.0.0 --port $PORT. Environment variables (SECRET_KEY, DATABASE_URL, FRONTEND_URL) are set in Render's dashboard, never committed to git.
Q: How does the frontend talk to the backend across different domains?
A: Through CORS (Cross-Origin Resource Sharing). The FastAPI backend has CORS middleware configured with the Vercel frontend URL as an allowed origin. Without this, the browser would block requests from vercel.app to render.com for security reasons. The FRONTEND_URL environment variable is comma-separated, so I can allow both the Vercel URL and localhost for development without code changes.
Q: What environment variables does this project use and why aren't they in git?
A: Key variables: SECRET_KEY (signs JWTs — if leaked, anyone can forge tokens), DATABASE_URL (database credentials), FRONTEND_URL (CORS allowed origin), and optionally OPENAI_API_KEY. These are never committed to git because they contain secrets. The repo has a .env.example file showing which variables are needed but with placeholder values. On Render and Vercel, these are set through the platform's secure environment variable UI and injected at build/runtime.

🧹 Code Quality & Patterns

Q: How did you organize the backend code?
A: I used the modular router pattern — each domain has its own file: auth_routes.py, tickets.py, comments.py, categories.py, users.py, ai_suggest.py. main.py is just the app initialization and router registration — about 56 lines. This keeps each file focused and under 200 lines, making it easy to navigate and test. Pydantic schemas are all centralized in schemas.py so input/output contracts are in one place.
Q: How did you handle errors in the API?
A: FastAPI handles validation errors automatically — if a request body doesn't match the Pydantic schema, it returns a structured 422 response. For business logic errors, I raise HTTPException with appropriate status codes: 404 for "ticket not found", 403 for "access denied", 400 for "invalid status transition", 401 for auth failures. This makes the API predictable for the frontend — it always receives a JSON error response with a detail field.
Q: What is the get_db dependency and why use it?
A: get_db is a FastAPI dependency that creates a new SQLAlchemy database session for each request and closes it afterward, even if an exception occurs: 
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
This is the dependency injection pattern. Instead of creating sessions manually in each endpoint, I declare db: Session = Depends(get_db) and FastAPI handles the session lifecycle. It prevents connection leaks and keeps endpoint code clean.

💬 Behavioral / HR Questions

Q: What was the hardest part of building this project?
A: The trickiest part was getting the authentication flow completely right between the frontend and backend. The JWT token needs to be attached to every API request, but it also needs to be validated on startup (so the user stays logged in on page refresh) and cleared on expiry (so stale tokens don't cause silent failures). I solved this with two things: an Axios interceptor that auto-attaches the token to every request header, and an AuthContext that on app load calls GET /api/auth/me — if it returns 401, the stored token is invalid and the user is sent to the login page.
Q: What would you do differently if you built this again?
A: A few things. First, I'd store the JWT in an HttpOnly cookie instead of localStorage for better XSS protection. Second, I'd add a proper test suite — at minimum, pytest integration tests for each API endpoint. Third, I'd use Alembic for database migrations instead of create_all() on startup, so schema changes in production are tracked and reversible. Fourth, I'd add WebSocket support for real-time comment updates so the agent doesn't need to refresh the page to see new customer replies.
Q: How long did this project take you?
A: I built this as a learning exercise over a few days. The backend took the most time to design correctly — I wanted the schema and API contracts to be solid before building the frontend on top of them. The frontend was faster because once the API was defined, the React components were mostly data-fetching + rendering. Deployment was surprisingly smooth because both Vercel and Render have good defaults for this stack.
Q: How does this project compare to Zendesk or Freshdesk?
A: It's a much simpler version — Zendesk has email ingestion, SLA tracking, knowledge bases, macros, integrations with 1000+ tools, and handles millions of tickets. What my project demonstrates is the core domain model and architecture that underpins those systems: ticket lifecycle management, role-based access, assignment workflows, threaded comments, and AI-assisted replies. Building this gave me an appreciation for the engineering complexity those products hide from end users.
Q: What would you add to make this production-ready for a real company?
A: Several things: Email notifications (notify customer when status changes via SendGrid), file attachments (let customers upload screenshots stored in S3), rate limiting on the API (prevent abuse), proper logging & monitoring (structured logs + Sentry for errors), database migrations (Alembic), automated tests (pytest + React Testing Library), CI/CD pipeline (GitHub Actions to run tests before deploy), and an admin role to manage agents and view system-wide analytics.
💡 Architecture Best Practice Stateless authentication (JWT) is chosen here so that the REST API can scale horizontally across multiple instances without a shared session store, keeping database queries focused on ticketing operations.
