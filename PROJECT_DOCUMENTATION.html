1. Project Overview & Motivation

Every SaaS company, e-commerce platform, and service provider needs a system to track customer issues вҖ” from initial report to resolution. I built this Customer Support Ticket System to solve that exact problem: a platform where customers submit support requests and agents manage, respond to, and resolve them.

What Problem Does This Solve?

  • Customers can create tickets, track their status, and communicate with support agents вҖ” all in one place
  • Support Agents get a centralized dashboard to manage all tickets, assign workload, and use AI-assisted replies
  • Organizations get visibility into support metrics вҖ” open/in-progress/resolved counts, response times

Key Features I Implemented

  • Role-based access control (Customer vs Agent) with distinct UI and permissions
  • Complete ticket lifecycle: Open вҶ’ In Progress вҶ’ Resolved with transition validation
  • JWT-based authentication with bcrypt password hashing
  • 16+ RESTful API endpoints with auto-generated Swagger documentation
  • AI-powered reply suggestions (template-based + optional OpenAI integration)
  • Category system, priority levels, search & filtering
  • Responsive SPA frontend with modern UI/UX
Why This Project? I chose to build a ticketing system because it mirrors real-world SaaS tools (Zendesk, Freshdesk, Jira Service Desk) and demonstrates full-stack competency: authentication, database design, REST API architecture, state management, and UI development вҖ” all in one project.

2. Approach & Architecture

Architecture Pattern: Client-Server with REST API

I followed a decoupled architecture вҖ” the React frontend and FastAPI backend are completely separate applications communicating via REST API. This approach offers:

  • Independent deployment: Frontend on Vercel CDN, backend on Render
  • Scalability: Each layer can scale independently
  • API reusability: The same backend can serve a mobile app or third-party integrations

4. Database Schema Design

I designed a normalized relational schema with four tables and indexed foreign-key relationships. The schema follows 3NF (Third Normal Form) to eliminate data redundancy.

Design Decisions

  • Two FK on tickets вҶ’ users: customer_id (who created it) and agent_id (who's handling it). This models the real-world assignment workflow.
  • Cascade delete on comments: When a ticket is deleted, its comments are automatically removed.
  • Indexed columns: status, priority, customer_id, and agent_id are indexed for fast filtering on the dashboard.
  • is_ai_generated flag: Comments created via AI suggestion are marked, providing transparency.
  • Soft delete pattern: is_active on users allows deactivation without data loss.

5. Backend Implementation

FastAPI Application Structure

The backend follows a modular router pattern. Each domain (auth, tickets, comments, etc.) has its own router file with dedicated endpoints. This keeps the codebase organized and each file under 200 lines.

# main.py вҖ” Application entry point
app = FastAPI(title="Customer Support Ticket System")

# CORS middleware for cross-origin requests from React
app.add_middleware(CORSMiddleware, allow_origins=[frontend_url], ...)

# Register all routers
app.include_router(auth_routes.router)    # /api/auth/*
app.include_router(tickets.router)        # /api/tickets/*
app.include_router(comments.router)       # /api/tickets/*/comments
app.include_router(categories.router)     # /api/categories/*
app.include_router(users.router)          # /api/users/*
app.include_router(ai_suggest.router)     # /api/ai/*

Request Validation with Pydantic

Every endpoint uses Pydantic schemas for input validation. FastAPI automatically returns 422 errors for invalid data:

class TicketCreate(BaseModel):
    title: str = Field(..., min_length=5, max_length=200)
    description: str = Field(..., min_length=10)
    priority: str = Field(default="medium", pattern="^(low|medium|high|urgent)$")
    category_id: Optional[int] = None

Status Transition Validation

I enforce valid ticket status transitions to prevent inconsistent states:

valid_transitions = {
    "open": ["in_progress", "resolved"],
    "in_progress": ["open", "resolved"],
    "resolved": ["open"],  # Can reopen resolved tickets
}

if new_status not in valid_transitions[current_status]:
    raise HTTPException(400, "Invalid status transition")
Auto-Generated API Docs FastAPI automatically generates interactive Swagger UI at /docs and ReDoc at /redoc. This means zero effort for API documentation вҖ” it's always up to date with the code.

6. Frontend Implementation

React SPA Architecture

The frontend is a Single Page Application built with React 18. Key patterns used:

  • React Router v6: Client-side routing with protected routes
  • Context API: Global auth state (user data, login/logout functions)
  • Axios with interceptors: Centralized API client with automatic JWT attachment and 401 handling
  • Component composition: Reusable components (StatusBadge, PriorityBadge, Navbar)

Authentication Flow

1. User submits login form
2. POST /api/auth/login вҶ’ receive JWT token
3. Store token in localStorage
4. GET /api/auth/me вҶ’ fetch user profile
5. Set user in AuthContext
6. Axios interceptor auto-attaches token to all requests
7. On 401 response вҶ’ clear token, redirect to /login

Protected Routes

// PrivateRoute component
function PrivateRoute({ children }) {
  const { user } = useAuth()
  if (!user) return <Navigate to="/login" />
  return children
}

// Usage in App.jsx
<Route path="/dashboard" element={
  <PrivateRoute><Dashboard /></PrivateRoute>
} />

UI/UX Decisions

  • Responsive design: Tailwind's responsive utilities ensure the app works on mobile and desktop
  • Loading states: Spinner shown during API calls to prevent confusion
  • Toast notifications: react-hot-toast for non-intrusive success/error feedback
  • Color-coded badges: Status (yellow/blue/green) and priority (gray/orange/red) are instantly recognizable
  • Split-panel login: Professional branding panel + form panel (desktop), form-only on mobile

Role-Based Access Control

# Dependency factory for role checking
def require_role(allowed_roles: list):
    def role_checker(current_user = Depends(get_current_user)):
        if current_user.role not in allowed_roles:
            raise HTTPException(403, "Forbidden")
        return current_user
    return role_checker

# Usage: Only agents can assign tickets

    ...

Security Measures

  • bcrypt hashing: Adaptive work factor protects against brute-force attacks
  • CORS configuration: Only configured origins can access the API
  • Input validation: Pydantic schemas reject malformed input before it reaches business logic
  • Row-level security: Customers can only see their own tickets (enforced in query filters)
  • No sensitive data in JWT: Token only contains user_id and role вҖ” not email or password

8. AI Reply Suggestion Engine

The AI suggestion feature helps agents draft replies faster. I implemented a dual-approach system:

Approach 1: Template-Based (Default)

Works out-of-the-box without any API keys. Templates are categorized by ticket category (billing, technical, account, general) and randomized to avoid repetition.

REPLY_TEMPLATES = {
    "billing": ["Thank you for reaching out about your billing concern...", ...],
    "technical": ["Thank you for reporting this technical issue...", ...],
    "account": ["I understand you're having trouble with your account...", ...],
    "general": ["Thank you for contacting our support team...", ...],
}

Approach 2: OpenAI Integration (Optional)

If an OPENAI_API_KEY is configured, the system constructs a context-aware prompt including the ticket title, description, category, priority, and recent comments, then sends it to GPT-3.5-turbo for a tailored response.

Graceful Fallback If the OpenAI API fails (rate limit, network error, invalid key), the system silently falls back to template-based suggestions. The user always gets a response.

11. Challenges Faced & Solutions

ChallengeSolution
N+1 query problem вҖ” Loading tickets with customer/agent names caused hundreds of SQL queries Used SQLAlchemy's joinedload to eagerly fetch related data in a single query. Dashboard load time dropped from ~800ms to ~50ms.
JWT token management вҖ” Keeping auth state synced between API client and React context Axios interceptors handle token attachment and 401 cleanup. AuthContext initializes by calling /api/auth/me on app load to validate stored tokens.
Status transition validation вҖ” Preventing invalid state changes (e.g., resolved вҶ’ in_progress directly) Defined a transition map and validated before each update. Returns clear error messages for invalid transitions.
CORS configuration вҖ” React dev server on port 5173, FastAPI on 8000 Configured CORS middleware with the frontend URL. Used Vite's proxy in development for seamless API calls.
Role-based UI вҖ” Same app, different capabilities for customers vs agents AuthContext exposes user.role. Components conditionally render features (e.g., "Assign to me" button only for agents). Backend enforces access at the API level too.
Database portability вҖ” Need SQLite for easy setup, PostgreSQL for production SQLAlchemy ORM abstracts the database. Only the connection string changes between SQLite and PostgreSQL.

Tech stack: React + Tailwind on the frontend (deployed to Vercel), FastAPI + SQLAlchemy on the backend (deployed to Render), and SQLite locally / can use PostgreSQL in production. Authentication is JWT-based with bcrypt password hashing. The whole system has role-based access control вҖ” customers and agents see completely different UIs even though it's one app.
Q: Why did you build this specific project?
A: I wanted one project that demonstrates everything a full-stack role requires: database design, REST API architecture, authentication, role-based authorization, state management in React, and cloud deployment. A ticketing system is perfect because it has a real-world business context вҖ” companies like Zendesk and Freshdesk make millions solving exactly this problem. рҹҸ—пёҸ Architecture & Design

Q: Why did you separate the frontend and backend? Why not a monolith?
A: Decoupled architecture gives three main advantages: independent deployment вҖ” I can push a frontend fix without restarting the backend; scalability вҖ” if the API gets heavy traffic, I can scale just the backend; and API reusability вҖ” if we later build a mobile app, it consumes the same FastAPI backend without any changes. The trade-off is more complexity in CORS configuration and separate deployment pipelines, but for any real-world SaaS product, this is the standard approach.
Q: How did you design the database schema?
A: I followed 3NF (Third Normal Form) normalization. There are four tables: users, categories, tickets, and comments. The key design decision was putting two foreign keys on the tickets table referencing users вҖ” one for customer_id (who created it) and one for agent_id (who's handling it). This models the real-world assignment workflow cleanly. I also added database indexes on status, priority, customer_id, and agent_id because those are the most-filtered columns on the dashboard query.
Q: What is the ticket lifecycle? How do status transitions work?
A: Tickets follow Open вҶ’ In Progress вҶ’ Resolved, with the option to reopen a resolved ticket back to Open. I enforce this in the backend with a transition map: 
valid_transitions = {
    "open":        ["in_progress", "resolved"],
    "in_progress": ["open", "resolved"],
    "resolved":    ["open"],  # can reopen
}
If the new status isn't in the allowed list, the API returns HTTP 400 with a clear error message. This prevents data inconsistencies вҖ” for example, you can't jump from open directly to resolved without the agent picking it up (though I do allow it for flexibility). The status change also auto-promotes a ticket to in_progress when an agent assigns it.

вҡҷпёҸ Tech Stack Decisions

Q: Why FastAPI and not Flask or Django?
A: Three reasons. First, FastAPI gives you automatic Swagger UI at /docs вҖ” zero extra work for API documentation. Second, it uses Pydantic for request validation, so I define the schema once and get type checking, serialization, and auto-generated docs all from the same class. Third, FastAPI is async-capable (ASGI), so it's more performant than Flask (WSGI) for I/O-bound workloads like database queries. Django would have been overkill вҖ” it comes with an ORM, admin, templating engine, and auth system I'd be replacing anyway.
Q: Why SQLAlchemy? Why not raw SQL or another ORM?
A: SQLAlchemy is Python's most mature ORM and abstracts the database engine, so my code runs on both SQLite (development) and PostgreSQL (production) with only a connection-string change. The most important feature I used was joinedload to solve the N+1 query problem вҖ” without it, loading 20 tickets with customer names and agent names would fire 41 separate SQL queries. With joinedload, it's a single JOIN query.
Q: Why React? Why not plain HTML/JS or Vue?
A: The dashboard has dynamic state вҖ” filters, live ticket updates, role-based conditional rendering, and form state management. React's component model and virtual DOM make managing this complexity clean and predictable. Context API handles global auth state without needing Redux for an app of this size. React also has the largest ecosystem and is the most-asked-for skill in frontend job listings, so it was the logical choice for a portfolio project.
Q: Why Vite instead of Create React App?
A: Vite's dev server starts in under a second and uses native ES modules, giving instant Hot Module Replacement. Create React App uses Webpack, which has 30вҖ“60 second cold starts on larger projects. For development productivity, Vite is simply better. CRA is also no longer actively maintained by the React team.

рҹ”җ Authentication & Security

Q: How does JWT authentication work in your system?
A: When a user logs in, the backend verifies their credentials, then issues a signed JWT token containing the user's ID, role, and expiry time. The frontend stores this token in localStorage and attaches it to every API request via an Axios interceptor in the Authorization: Bearer <token> header. On each protected endpoint, FastAPI's get_current_user dependency decodes and verifies the token. If it's expired or invalid, the API returns 401, and the Axios interceptor clears the stored token and redirects the user to the login page.
Q: Why JWT instead of session-based auth?
A: JWT is stateless вҖ” the server doesn't store session data in memory or a database. This is important for scalability: if I had multiple backend instances behind a load balancer, session-based auth would require a shared session store (like Redis). With JWT, any instance can verify the token by checking the cryptographic signature. The trade-off is that JWTs can't be invalidated before expiry вҖ” but I mitigated this by setting a short 60-minute expiry.
Q: Is storing JWT in localStorage safe?
A: This is a legitimate security question. localStorage is vulnerable to XSS attacks вҖ” if an attacker injects malicious JavaScript, they can read the token. The more secure alternative is storing the JWT in an HttpOnly cookie, which JavaScript can't access. For this project, localStorage was acceptable because: (1) it's a demo/portfolio project without real sensitive data, (2) the API validates tokens on every request, and (3) the CORS policy restricts which origins can call the API. In a real production app, I'd use HttpOnly cookies.
Q: How does role-based access work?
A: There are two roles: customer and agent. Enforcement happens at two levels. At the API level, I use a require_role() dependency factory вҖ” endpoints like "assign ticket" or "AI suggest" only accept requests from users with the agent role; otherwise they return HTTP 403. At the data level, customers can only query their own tickets вҖ” the list_tickets endpoint filters by customer_id == current_user.id for non-agents. Frontend-side, the user.role from AuthContext conditionally shows/hides UI elements, but I never rely solely on the frontend for security.
Q: How are passwords stored?
A: Passwords are hashed using bcrypt before being stored. bcrypt is an adaptive hashing algorithm вҖ” it has a configurable "work factor" that makes it intentionally slow to compute, which protects against brute-force attacks. Even if the database is compromised, the attacker can't reverse the hash. Plaintext passwords are never logged or stored anywhere in the system.

вҡЎ Performance & Scalability

Q: What is the N+1 query problem and how did you solve it?
A: The N+1 problem occurs when you run 1 query to fetch a list, then N additional queries to fetch related data for each item. For example, loading 20 tickets and then querying the customer name for each ticket = 21 queries. I solved this using SQLAlchemy's joinedload(): 
db.query(Ticket).options(
    joinedload(Ticket.customer),
    joinedload(Ticket.agent),
    joinedload(Ticket.category),
)
This generates a single SQL query with JOINs. The dashboard load time went from ~800ms to ~50ms as a result.
Q: How does pagination work?
A: The GET /api/tickets/ endpoint accepts page and limit query parameters. Pagination is implemented with SQL OFFSET and LIMIT: 
query.offset((page - 1) * limit).limit(limit).all()
The default is 20 tickets per page with a maximum of 100. This prevents loading thousands of rows into memory on a large dataset. The frontend sends ?page=1&limit=20 and can request the next page by incrementing the page number.
Q: What database indexes did you add and why?
A: I indexed status, priority, customer_id, and agent_id on the tickets table. These are the columns used in the most common WHERE clauses: "show all open tickets", "show my assigned tickets", "filter by priority". Without indexes, each of these would require a full table scan. With indexes, the database can look up matching rows in O(log n) time using a B-tree structure. Email on users is also indexed for fast login lookups.

рҹӨ– AI Feature

Q: Tell me about the AI reply suggestion feature.
A: When an agent is viewing a ticket, they can click "Suggest Reply" which calls POST /api/ai/suggest-reply. The system works in two modes. By default it uses a template-based system вҖ” it looks at the ticket's category (billing, technical, account, general) and returns a contextually appropriate pre-written response, randomized from a pool to avoid repetition. If an OPENAI_API_KEY is configured as an environment variable, it switches to GPT-3.5-turbo вҖ” it constructs a prompt with the ticket title, description, category, priority, and recent comments, then calls the OpenAI API. If the OpenAI call fails for any reason (rate limit, network error), it silently falls back to templates, so the feature always works.
Q: Why template-based and not always OpenAI?
A: Cost and reliability. OpenAI API calls cost money and have rate limits. For a demo or small deployment, templates provide instant, free suggestions that work offline. It also makes the project deployable without needing anyone to have or pay for an API key. The system is designed so upgrading to real AI is just adding an environment variable вҖ” no code changes needed.

A: Key variables: SECRET_KEY (signs JWTs вҖ” if leaked, anyone can forge tokens), DATABASE_URL (database credentials), FRONTEND_URL (CORS allowed origin), and optionally OPENAI_API_KEY. These are never committed to git because they contain secrets. The repo has a .env.example file showing which variables are needed but with placeholder values. On Render and Vercel, these are set through the platform's secure environment variable UI and injected at build/runtime.

рҹ§№ Code Quality & Patterns

Q: How did you organize the backend code?
A: I used the modular router pattern вҖ” each domain has its own file: auth_routes.py, tickets.py, comments.py, categories.py, users.py, ai_suggest.py. main.py is just the app initialization and router registration вҖ” about 56 lines. This keeps each file focused and under 200 lines, making it easy to navigate and test. Pydantic schemas are all centralized in schemas.py so input/output contracts are in one place.
Q: How did you handle errors in the API?
A: FastAPI handles validation errors automatically вҖ” if a request body doesn't match the Pydantic schema, it returns a structured 422 response. For business logic errors, I raise HTTPException with appropriate status codes: 404 for "ticket not found", 403 for "access denied", 400 for "invalid status transition", 401 for auth failures. This makes the API predictable for the frontend вҖ” it always receives a JSON error response with a detail field.
Q: What is the get_db dependency and why use it?
A: get_db is a FastAPI dependency that creates a new SQLAlchemy database session for each request and closes it afterward, even if an exception occurs: 
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
This is the dependency injection pattern. Instead of creating sessions manually in each endpoint, I declare db: Session = Depends(get_db) and FastAPI handles the session lifecycle. It prevents connection leaks and keeps endpoint code clean.

I solved this with two things: an Axios interceptor that auto-attaches the token to every request header, and an AuthContext that on app load calls GET /api/auth/me вҖ” if it returns 401, the stored token is invalid and the user is sent to the login page.
